**Researched:** 2026-04-14
**Domain:** CSS variable theming — palette migration, dual accent system, neon border treatment

## Summary

This is a pure CSS variable value swap. The theming infrastructure (CSS variables, classList theme toggle, Mantine v6 overrides) was fully built in Phases 1 and 3. This phase updates the VALUES inside those variables — no new mechanisms, no new classes, no layout changes.

The only structural addition is two new CSS variables (`--accent-secondary`, `--accent-secondary-glow`) to support the dual accent system (purple for buttons/actions, cyan for progress/status). All existing component classes (`monolith-card`, `btn-accent`, `progress-fill`, etc.) consume the variables and will pick up the new palette automatically.

The main risk area is hardcoded `rgba(59, 74, 107, ...)` values scattered directly in `App.css` hover/glow rules — these are NOT driven by variables and must be found and replaced individually. The planner must enumerate every occurrence.

**Primary recommendation:** Update `index.css` variable blocks first (both themes), then hunt down hardcoded rgba(59,74,107,...) in `App.css` and replace with new values. Add `--accent-secondary` / `--accent-secondary-glow` and update `progress-fill` to use them.

## Common Pitfalls

### Pitfall 1: Hardcoded rgba Values in App.css
**What goes wrong:** Variable swap looks complete but hover/glow effects still show old steel-blue color.
**Why it happens:** Several rules in App.css use literal `rgba(59, 74, 107, ...)` instead of `var(--accent-glow)`. These bypass the variable system entirely.
**How to avoid:** Before writing any CSS, grep the entire codebase for `rgba(59` and `rgba(59,` to enumerate every occurrence.
**Warning signs:** Buttons glow blue-gray on hover after the palette update.

Occurrences found in App.css [VERIFIED: codebase read]:
- Line 59: `.btn-accent:hover` — `0 0 12px rgba(59, 74, 107, 0.3)`
- Line 65: `.btn-accent:active` — `0 0 12px rgba(59, 74, 107, 0.15)`
- Line 396: `.mantine-Button-root:hover` — `0 0 12px rgba(59, 74, 107, 0.3)`
- Line 401: `.mantine-Button-root:active` — `0 0 12px rgba(59, 74, 107, 0.15)`

### Pitfall 2: Forgetting Progress Bar Dual Accent
**What goes wrong:** Progress bar remains purple after palette update, instead of cyan.
**Why it happens:** `progress-fill` references `var(--accent)` and `.mantine-Progress-bar` overrides to `var(--accent)`. Without an explicit secondary variable and targeted rule update, progress inherits primary accent.
**How to avoid:** Add `--accent-secondary` + `--accent-secondary-glow` to both theme blocks. Update `.progress-fill` to `background: var(--accent-secondary)` and `box-shadow: 0 0 8px rgba(0,200,255,0.5)`. Update `.mantine-Progress-bar` override to `var(--accent-secondary)`.

### Pitfall 3: Light Mode Contrast Regression
**What goes wrong:** WCAG AA compliance breaks for text-on-card with new navy text (#1A1A2E) on light lavender card (#E4E7F0).
**Why it happens:** #1A1A2E on #E4E7F0 is approximately 11:1 — that passes. But muted text rgba(26,26,46,0.60) on #E4E7F0 needs verification. At 60% opacity on a ~#E4E7F0 surface, effective color is ~#787898, which gives roughly 4.8:1 against white — marginal.
**How to avoid:** Verify muted text contrast in both modes after update. Phase 3 already confirmed dark muted text at 50% opacity passes (5.37:1). Light mode at 60% should be checked against card backgrounds, not just page bg.
**Warning signs:** Muted tag text or table headers become hard to read in light mode.

### Pitfall 4: `--bg-secondary` Light Mode Gap
**What goes wrong:** A light mode secondary background area uses the old warm ivory tone, visually breaking the cool lavender palette.
**Why it happens:** D-10 defines bg-primary, card, and card-hover but not `--bg-secondary` for light mode explicitly. Current value is #EAE6DF (warm gray).
**How to avoid:** Planner should choose a cool lavender-gray mid-tone as `--bg-secondary` light (discretion allowed per CONTEXT.md). Suggested: #E8EAF2 (between bg-primary #F0F2F8 and card #E4E7F0).

### Pitfall 5: Support Banner Orange Contrast Against New Backgrounds
**What goes wrong:** Orange banner text (#ffb74d dark / #bf360c light) that passed AA against old backgrounds may need re-verification against new navy/lavender backgrounds.
**Why it happens:** Banner background is rgba(255,183,77,0.08) against the page bg. New page bg is darker navy — may shift the effective composite color.
**How to avoid:** Verify `--banner-text` values pass WCAG AA after the page bg shifts. Generally navy backgrounds improve orange text contrast, so this is likely safe but worth confirming.

   - What's unclear: Is a purple spinner correct, or should it be cyan (progress/status = cyan per D-06)?
