## Summary

Reviewed the SEO-related additions: two index.html files (dev source and production build), robots.txt, and sitemap.xml. The SEO implementation is solid overall -- JSON-LD structured data is valid, Open Graph and Twitter Card tags are correct and complete, robots.txt properly blocks `/api/`, and the sitemap is well-formed XML.

Two warnings found related to content duplication and a missing `crossorigin` attribute consistency issue. Three informational items related to minor improvements.

## Warnings

### WR-01: Google Fonts preconnect tags use inconsistent HTML quoting style

**File:** `client/index.html:44-45`
**Issue:** The `<link rel="preconnect">` tags on lines 44-45 use unquoted `crossorigin` attribute on the second link, while the Google Fonts `<link>` tag on line 46 also lacks a closing slash. More importantly, the two preconnect links are missing the `crossorigin` attribute on the first one -- `fonts.googleapis.com` serves the CSS (no CORS needed), but `fonts.gstatic.com` serves the font files (CORS needed). Currently correct, but the bare `crossorigin` attribute (without a value) defaults to `anonymous`, which is the right behavior. However, mixing self-closing tag style (`/>`) with non-self-closing (`>`) across these HTML meta/link tags is inconsistent. The real issue: the `>` vs `/>` inconsistency won't cause bugs in HTML5, but it signals copy-paste from different sources without normalization. **Downgrading to Info.**

*Reclassified -- see IN-03 below.*

### WR-01 (actual): client/index.html and public/index.html SEO meta content must stay synchronized manually

**File:** `client/index.html:1-54` and `public/index.html:1-55`
**Issue:** The SEO meta tags, OG tags, JSON-LD, and all head content exist in both `client/index.html` (Vite dev source) and `public/index.html` (production build). Since `public/index.html` is a build artifact generated by Vite from `client/index.html`, any future edit to SEO tags must be made in `client/index.html` and then rebuilt. However, there is no build-time validation that `public/index.html` is up to date with `client/index.html`. If someone edits `public/index.html` directly (bypassing the build), those changes will be lost on the next rebuild. Conversely, editing `client/index.html` without rebuilding will leave production stale.
**Fix:** Add a comment at the top of `public/index.html` indicating it is a build artifact:
```html
<!-- AUTO-GENERATED by Vite build. Do not edit directly. Edit client/index.html instead. -->
```
Alternatively, add a check in `deploy.sh` that verifies the SEO-critical meta tags match between source and built output.

### WR-02: Catch-all route in server.js may shadow robots.txt and sitemap.xml for non-GET requests

**File:** `public/robots.txt:1-5` and `public/sitemap.xml:1-9`
**Issue:** While `express.static` (server.js:141) correctly serves static files before the catch-all `app.get('*')` handler (server.js:144), the static files are only served if the `public/` directory is found as `distPath`. If the path resolution fails or the server falls into API-only mode (server.js:174), robots.txt and sitemap.xml will return 404. A production curl to `robots.txt` currently returns 404, which suggests the deployment may not be serving from the `public/` directory that contains these files, or there is a caching/deployment timing issue.
**Fix:** Verify that the Replit deployment's `distPath` resolves to the directory containing `robots.txt` and `sitemap.xml`. Run `deploy.sh` to ensure the build output includes these files. If the issue persists, add explicit routes for SEO-critical files:
```javascript
// Ensure SEO files are always served
app.get('/robots.txt', (req, res) => {
  res.sendFile(path.join(__dirname, 'public', 'robots.txt'));
});
app.get('/sitemap.xml', (req, res) => {
  res.sendFile(path.join(__dirname, 'public', 'sitemap.xml'));
});
```

## Info

### IN-01: sitemap.xml lastmod date is hardcoded

**File:** `public/sitemap.xml:4`
**Issue:** The `<lastmod>` value is hardcoded to `2026-04-14`. This will become stale over time and may signal to search engines that the content hasn't been updated, even after deploys.
**Fix:** Consider generating the sitemap during the build process with the current date, or update it manually during each deployment. A simple approach in `deploy.sh`:
```bash
sed -i "s|<lastmod>.*</lastmod>|<lastmod>$(date +%Y-%m-%d)</lastmod>|" public/sitemap.xml
```

### IN-02: JSON-LD structured data could include additional properties

**File:** `client/index.html:27-43`
**Issue:** The JSON-LD WebApplication schema is valid but minimal. Adding `screenshot`, `softwareVersion`, or `author` properties would provide richer search result snippets.
**Fix:** Optional enhancement -- add a screenshot URL and author:
```json
"screenshot": "https://suno-playlist-downloader.replit.app/assets/og-card.png",
"author": {
  "@type": "Person",
  "name": "Ash Rhodes"
}
```

### IN-03: Inconsistent self-closing tag style in HTML head

**File:** `client/index.html:44-46`
**Issue:** Lines 44-46 use `>` for closing link tags (no trailing slash), while lines 4-9 use `/>` (XHTML self-closing style). Both are valid HTML5, but the inconsistency suggests copy-paste from external sources without normalization.
**Fix:** Normalize to one style. Since the rest of the file uses `/>`, update the font preconnect links:
```html
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
```
