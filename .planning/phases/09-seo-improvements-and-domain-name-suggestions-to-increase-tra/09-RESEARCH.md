## Summary

Phase 9 is a pure HTML/static-file phase. All changes live in `client/src/index.html` (source of truth) and `public/` (deployed output). No React, no JavaScript changes — just `<head>` tag additions and two new static files.

The codebase is a Vite-built SPA served by Express `express.static('public/')`. Because Express serves `public/` as the static root, dropping `robots.txt` and `sitemap.xml` directly in `public/` is sufficient — no server.js route changes required. Vite does not copy files from `public/` root automatically; the build process is `client/` → `client/dist/` → manually copied to `public/` via `deploy.sh`. Static files placed in `public/` must stay there across deploys (they are not overwritten by `deploy.sh` which only `cp -r client/dist/* public/`).

The nanobanana OG image (D-02, D-11) must be generated at orchestrator level — confirmed from project memory. The planner should make this a Wave 0 blocking task.

**Primary recommendation:** Add all meta/structured-data to `client/src/index.html`, copy to `public/index.html` manually, place robots.txt and sitemap.xml in `public/`, generate OG image via nanobanana at orchestrator level.

### Pattern 2: Static files in public/ survive deploy.sh

**What:** `deploy.sh` does `cp -r client/dist/* public/` — this copies Vite build output into `public/`. Files already in `public/` that do NOT exist in `client/dist/` (e.g., `robots.txt`, `sitemap.xml`, `assets/og-card.png`) are NOT deleted.

**Verification:** `deploy.sh` uses `cp -r ... public/` not `rm -rf public/` first, so new static files placed directly in `public/` persist across future deploys. Confirmed by reading deploy.sh. [VERIFIED: deploy.sh line 9-10]

### Pattern 3: robots.txt pointing to sitemap

```
# Source: Google robots.txt guidelines
User-agent: *
Allow: /

Sitemap: https://your-domain.com/sitemap.xml
```

Optional — block AI training crawlers (2025 best practice if desired):
```
User-agent: GPTBot
Disallow: /

User-agent: *
Allow: /
```

Blocking AI bots is Claude's discretion per decisions.

### Pattern 4: SPA sitemap.xml (single URL)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://your-domain.com/</loc>
    <lastmod>2026-04-14</lastmod>
    <changefreq>monthly</changefreq>
    <priority>1.0</priority>
  </url>
</urlset>
```

### Anti-Patterns to Avoid

- **Editing only `public/index.html` directly:** The next `deploy.sh` run will overwrite it from Vite build output, which comes from `client/src/index.html`. Always edit the source file.
- **Absolute URLs with `localhost`:** og:url, og:image, and canonical must use the live domain, not localhost. Use the actual Replit deployment URL or custom domain.
- **Using `WebApplication` vs `SoftwareApplication`:** Google's rich results for software apps work best with `WebApplication` subtype since this runs in a browser. `SoftwareApplication` is the parent type — both work, but `WebApplication` is more specific. [CITED: schema.org/SoftwareApplication]
- **Blocking /assets/ in robots.txt:** Google needs CSS/JS/image files to render pages. Never disallow /assets/.

## Common Pitfalls

### Pitfall 1: Editing public/index.html without editing client/src/index.html

**What goes wrong:** Meta tags appear in production, then disappear after the next `yarn build` + `deploy.sh` run.
**Why it happens:** deploy.sh overwrites public/index.html from Vite dist output. Vite builds from client/src/index.html.
**How to avoid:** ALWAYS edit client/src/index.html first. Then manually sync the `<head>` changes to public/index.html for immediate effect without a full build.
**Warning signs:** SEO tags present in public/ but missing from client/src/.

### Pitfall 2: og:image with a relative URL

**What goes wrong:** Social media crawlers (Facebook, Twitter/X, LinkedIn) can't resolve relative paths. The preview shows no image.
**Why it happens:** Crawlers fetch HTML from the root URL, but don't resolve relative paths the same way browsers do.
**How to avoid:** Use fully qualified absolute URL: `https://your-domain.com/assets/og-card.png`
**Warning signs:** Social card previews on opengraph.xyz showing broken image.

### Pitfall 3: canonical URL pointing to localhost

**What goes wrong:** Google indexes localhost as the canonical — effectively de-indexing the real site.
**Why it happens:** Developer copies meta tags from local dev without updating the domain.
**How to avoid:** Canonical and og:url must use the live Replit URL (or custom domain).

### Pitfall 4: robots.txt blocking /assets/

**What goes wrong:** Google can't render the page — CSS and JS are blocked. Page indexed with no styling context.
**Why it happens:** Over-aggressive disallow rules.
**How to avoid:** robots.txt should only `Allow: /` with no specific Disallow entries (or disallow only `/api/`).

### Pitfall 5: JSON-LD missing `offers.price`

**What goes wrong:** Google validation rejects the structured data — no rich results eligibility.
**Why it happens:** Google requires `offers` with a `price` field (even `"0"` for free apps).
**How to avoid:** Always include `"offers": { "@type": "Offer", "price": "0", "priceCurrency": "USD" }`.
[CITED: developers.google.com/search/docs/appearance/structured-data/software-app]
