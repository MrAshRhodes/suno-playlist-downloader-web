### Dark Mode (`:root.dark-mode`)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0A0A0A` | Page background (`--bg-primary`) |
| Secondary (30%) | `#1A1A1A` | Card surfaces (`--bg-card`) |
| Accent (10%) | `#3B4A6B` | Buttons, focus rings, progress bar, badges, logo tint, links, active states |
| Destructive | `#C0392B` | Error status icons only (no destructive actions in Phase 1) |

Accent reserved for: primary action buttons (`btn-accent`), input focus rings, Mantine Progress bar, Mantine Badge filled variant, theme toggle active indicator, active download row marker.

Accent states:
- Default: `#3B4A6B`
- Hover: `#4E5F80` (no glow — Phase 3 concern)
- Active/pressed: `#2F3D58`

## Card Contract

All content sections receive uniform card treatment — no tiered elevation system.

```css
/* Rename .glass-card → .monolith-card in App.css */
/* Remove backdrop-filter — Monolith uses solid surfaces, not glass */
.monolith-card {
  background: var(--bg-card);
  border: 1px solid var(--border-color);
  border-radius: 24px;
  box-shadow: var(--shadow-card);
  transition: background-color 0.3s ease, border-color 0.3s ease, box-shadow 0.3s ease;
}
```

Rename touch-point: `.glass-card` used in `App.tsx` line 269. One rename required.

Affected sections: song table, URL input area, info banner, support banner — all identical radius and shadow. No exceptions.

**Source:** CONTEXT.md D-01, D-02, D-03. RESEARCH.md Pattern 3.

## Theme Transition Contract

Transition duration: 0.3s ease on concrete CSS properties only. CSS custom properties are not transitionable — the transition must be declared on `background-color`, `color`, `border-color`, `box-shadow` directly.

Body background cross-fade: requires deleting `document.body.style.backgroundColor` inline style injection in `main.tsx` AppWrapper (lines 30-36). The `App.css` body rule with `!important` handles this correctly without JS.

Theme toggle mechanism: classList on `<html>` (`dark-mode` / `light-mode`). `useDarkMode.ts` hook — no changes needed.

No animated overlay. No JS-driven transition. CSS only.

**Source:** CONTEXT.md D-10, D-11. RESEARCH.md Pitfall 1, Pitfall 2, Pitfall 3.

## Component Overrides (Mantine v6)

Mantine v6 ignores CSS vars for its internal component colors. These targeted overrides must be added to `App.css`:

```css
/* Button — filled variant */
.mantine-Button-root[data-variant="filled"] {
  background-color: var(--accent) !important;
}
.mantine-Button-root[data-variant="filled"]:hover {
  background-color: var(--accent-hover) !important;
}
.mantine-Button-root[data-variant="filled"]:active {
  background-color: var(--accent-active) !important;
}

/* Progress bar */
.mantine-Progress-bar {
  background-color: var(--accent) !important;
}

/* Badge */
.mantine-Badge-root[data-variant="filled"] {
  background-color: var(--accent) !important;
}
```

Apply only where Mantine components are actually rendered (audit `App.tsx` imports vs render output first).

**Source:** RESEARCH.md Pitfall 4. RESEARCH.md Open Question 1.

## Logo Contract

Generation method: nanobanana MCP + ImageMagick chromakey removal.

Prompt direction (Claude's discretion per D-09): minimalist vinyl record or abstract waveform icon. Chromakey green background (#00FF00). Flat design. White icon. 256x256 output.

Removal command:
```bash
magick /tmp/logo-raw.png -fuzz 15% -transparent "#00FF00" client/public/assets/logo.png
```

Fallback if ImageMagick unavailable: retain existing `<IconVinyl>` Tabler icon, styled with Deep Blue (`var(--accent)`) color. Do not block phase on logo.

Output path: `client/public/assets/logo.png`
Usage: Replace icon in `App.tsx` header with `<img src="/assets/logo.png" width="38" height="38" alt="Suno Downloader" />`

**Source:** CONTEXT.md D-09, CLAUDE.md ImageMagick workflow.

## Interaction States (Phase 1 Scope)

Phase 1 delivers static visual state only. No animations, no glow effects — those are Phase 3.

| Element | State | Visual Treatment |
|---------|-------|-----------------|
| Button (accent) | default | `var(--accent)` background, white text |
| Button (accent) | hover | `var(--accent-hover)` background, no glow |
| Button (accent) | active/pressed | `var(--accent-active)` background |
| Button (accent) | disabled | `var(--accent)` at 45% opacity, not-allowed cursor |
| Input field | default | `var(--bg-input)` background, `var(--border-color)` border |
| Input field | focus | `var(--accent)` border color, 2px width |
| Card | default | `var(--bg-card)` + `var(--shadow-card)` |
| Card | hover | no hover state in Phase 1 (Phase 3 concern) |
| Theme toggle | both modes | `var(--accent)` active indicator |

Transition: `background-color 0.3s ease, color 0.3s ease, border-color 0.3s ease, box-shadow 0.3s ease` on all elements that change between modes.

**Source:** CONTEXT.md D-08. ROADMAP.md Phase 1 success criteria (no glow on hover — explicitly Phase 3).
